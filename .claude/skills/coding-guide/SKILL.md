# コード編集時の心構え
## 型を使う
TypeScriptの定義された型に沿って、素直にコードを書きます。  
必要であれば、自分でも型やインターフェースを定義してください。

## 既存の実装を参考に
既存の実装を探して参照したり、同じ考え方でコードを書きます。目的に合ったnpmパッケージがあればインストールして使います。

## ドキュメントを参照する
APIやライブラリのことを知るためにソースコードを読み込みたくなるのはわかりますが、まずはドキュメントや仕様書を参照したり思い出したりします。  
具体的には以下のようなものです。

- MDN
- IETF RFC
- AGENTS.md
- SKILL.md
- npmパッケージのREADME
- Cloudflareのドキュメント
- DeepWiki

## コストは最低限に
特にCloudflare WorkersはCPUやメモリが制限されています。

例えばファイルを扱うとき、ファイルのすべてをArrayBufferなどでメモリに載せるのは避け、Streamを駆使して細切れに扱うべきです。  
Cloudflare R2は読み出しが無料なので、積極的に使います。

Streamでも全てのチャンクを処理するとタイムアウトするので、上流とクライアントを直結するようにしてください。  
クライアントではもう少し使うリソースを増やしても良いですが、それでもStreamは積極的に使うようにします。

## 1ファイルの行数を適切に
1つのファイルに500行近いコードを書くとかなりやりすぎな感じがあります。  
特に複雑なロジックは、新しくファイルを作ると良いです。

## コメントを残そう
コメントまみれでも読みにくくなってしまいますが、次のような場面では積極的にコメントを残すようにしてください。

- 調べて初めて発見した、APIやnpmパッケージなどの仕様
- 直感に反する、トリッキーだが動作した実装
- 複雑な思考を経て修正できた箇所
- ワンライナーコードの解説

エージェントスキル `<project root>/.claude/skills/*/SKILL.md` に書き残すことも考えてください。  
例えば、`<project root>/.claude/skills/stream/SKILL.md`では、ECMAScriptのストリームに関する注意点が書かれています。

作業中のIssueやPRだけに関する事柄であれば、複雑だった箇所をIssue・PRコメントで残すと良いと思います。  
Issueには仕様についてのまとめ、PRには具体的なコードに関することを書くと良いです。

Agent MemoryなどClaudeで準備されているメモリ機能よりは、コード内コメント、GitHubコメントやSKILL.mdを使って、人間でもアクセスしやすいようにしてください。
